# Stack tecnológico del monorepo

One repo, four independent apps deployed to Vercel.

## Decisions per app

### CRM — Next.js 15 + Drizzle + Supabase

**Why Next.js App Router:** SSR + API routes in one repo. `requireAuth()` runs on the server before rendering. No client-server round trip for auth checks.

**Why Drizzle ORM:** TypeScript-first, generates types from schema. Schema is source of truth; all types are inferred. Lightweight vs Prisma.

**Why Supabase:** PostgreSQL + Auth + Storage + Row-Level Security in one managed service. No separate auth server. `tokenPortal` pattern works cleanly with Supabase SSR cookies.

**Why not a separate API?** The CRM is the only consumer of its own API (plus the two portals). A standalone API would add a network hop with no benefit.

### Landing — Astro 6

**Why Astro:** Static output, zero JS by default. Landing is mostly static content with one dynamic form (lead capture). Astro's island architecture handles that cleanly. No React bundle shipped to users.

### Cotizador — React + Vite

**Why standalone React app?** The cotizador is embedded as an iframe in the landing. Keeping it as a separate Vite build means it can also be embedded in any third-party site (Wix, etc.) without pulling in Astro.

**Why not an Astro component?** The cotizador has complex multi-step state that benefits from React hooks. Astro components are not ideal for heavy interactive flows.

### Chatbot (Fase 4) — Next.js 14 + Vercel AI SDK

Not implemented yet. Directory `chatbot/` exists with empty `src/`. See [[wiki/versions/v3]].

## Shared

- **Tailwind CSS 4** — all apps.
- **Vercel** — all apps hosted, cron jobs on CRM.
- **Supabase** — shared PostgreSQL instance, separate Auth contexts per portal.
